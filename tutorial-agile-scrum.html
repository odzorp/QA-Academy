What is Agile?

Agile is an iterative approach to project management and software development that helps teams deliver value to their customers faster and with fewer headaches. Instead of betting everything on a single launch, an agile team delivers work in small, consumable increments. Requirements, plans, and results are evaluated continuously so teams have a natural mechanism for responding to change quickly.

The Agile Manifesto, created in 2001 by 17 software developers, emphasizes individuals and interactions over processes and tools, working software over comprehensive documentation, customer collaboration over contract negotiation, and responding to change over following a plan.

Why Agile for QA Engineers?

  • Early Testing: QA is involved from sprint planning, catching bugs early
  • Continuous Feedback: Regular retrospectives improve testing processes
  • Adaptability: Test plans evolve with changing requirements
  • Collaboration: Close working relationship with developers and product owners
  • Incremental Delivery: Each sprint produces testable, shippable increments

What is Scrum?

Scrum is a lightweight framework that helps people, teams and organizations generate value through adaptive solutions for complex problems. It is founded on empiricism—the idea that knowledge comes from experience and decisions are based on observation. The three pillars of Scrum are Transparency, Inspection, and Adaptation.

🔍
Transparency
All aspects must be visible to those responsible for outcomes
📊
Inspection
Frequently inspect artifacts and progress toward goals
⚡
Adaptation
Adjust processes when inspection reveals deviation

Scrum Roles

Scrum defines three accountabilities that together create the Scrum Team:

Product Owner

The Product Owner is accountable for maximizing the value of the product resulting from the work of the Scrum Team. They manage the Product Backlog, prioritizing items based on business value, user needs, and technical dependencies.

Scrum Master

The Scrum Master is accountable for establishing Scrum as defined in the Scrum Guide. They act as a servant-leader for the Scrum Team, facilitating ceremonies, removing impediments, and coaching on Agile principles.

Developers

Developers are the people in the Scrum Team that are committed to creating any aspect of a usable Increment each Sprint. This includes QA engineers who are responsible for testing the increment.

The Scrum Events

Scrum uses events to create regularity and to minimize the need for meetings not defined in Scrum. All events are time-boxed to ensure productive discussions.

🏃

Sprint (2-4 weeks)

The heartbeat of Scrum. A fixed time-box during which a "Done", useable, and potentially releasable product increment is created.

📋

Sprint Planning

Time-boxed to 8 hours for a one-month sprint. The work to be performed in the Sprint is planned at Sprint Planning.

🗓️

Daily Scrum

15-minute event for Developers to plan work for the next 24 hours. QA engineers share testing progress and any blockers.

🔍

Sprint Review

Held at the end of the Sprint to inspect the Increment and adapt the Product Backlog. Demo the work to stakeholders.

💭

Sprint Retrospective

Opportunity for the Scrum Team to inspect itself and create a plan for improvements during the next Sprint.

The Scrum Artifacts

Scrum's artifacts represent work or value. They are designed to maximize transparency of key information.

📝
Product Backlog
An ordered list of everything that is known to be needed in the product. Always evolving and never complete.
✅
Sprint Backlog
The set of Product Backlog items selected for the Sprint plus a plan for delivering them.
🚀
Increment
The sum of all Product Backlog items completed during a Sprint and prior Sprints.

Definition of Done

The Definition of Done (DoD) is a shared understanding of what it means for work to be complete. For QA engineers, the DoD typically includes:

# Example Definition of Done for a User Story Acceptance Criteria: ✓ All automated tests pass ✓ Code has been peer reviewed ✓ Documentation updated ✓ Manual test cases executed ✓ No critical or high severity bugs open ✓ Performance tests completed (if applicable) ✓ Security scan passed (if applicable) ✓ Product Owner accepted the story

User Stories and Acceptance Criteria

User Stories are short, simple descriptions of a feature told from the perspective of the person who wants the new capability. They follow the format: "As a [type of user], I want [goal] so that [benefit]."

INVEST in Good User Stories

  • Idependent - Can be delivered independently
  • Negotiable - Not a contract, details can be discussed
  • Valuable - Delivers value to users
  • Estimable - Team can estimate the effort
  • Small - Can be completed in one Sprint
  • Testable - We know when it's done
# Example User Story with Acceptance Criteria User Story: As a registered user, I want to reset my password so that I can regain access to my account if I forget it. Acceptance Criteria: 1. User can request password reset via email 2. Reset link expires after 24 hours 3. User must confirm new password twice 4. Confirmation email sent after successful reset 5. User logged out of all devices after reset

Writing Test Cases from User Stories

QA engineers play a crucial role in validating user stories. Here's how to approach testing:

1. Review Acceptance Criteria

Before writing any test cases, thoroughly review the acceptance criteria. If anything is unclear, ask the Product Owner for clarification. Never assume requirements.

2. Identify Test Scenarios

Break down each acceptance criterion into testable scenarios. Consider positive and negative test cases, boundary conditions, and edge cases.

3. Create Test Cases

# Example Test Case Format Test Case ID: TC-PW-001 Title: Password Reset - Valid Email Pre-condition: User is registered with valid email Steps: 1. Navigate to login page 2. Click "Forgot Password" 3. Enter registered email 4. Click "Submit" Expected Result: - Success message displayed - Email received with reset link

4. Execute and Report

Run your test cases and document results. Report any defects found using your chosen bug tracking tool. Ensure defects are linked to the relevant user story.

Jira Workflow for QA

Jira is the most popular tool for managing Agile projects. Here's how to set up a QA-focused workflow:

# Jira Workflow for QA Teams Standard Issue Transitions: To Do → In Progress (Developer starts work) In Progress → In Review (Developer submits for review) In Review → In QA (Ready for testing) In QA → In Progress (QA finds issues, returns to Dev) In QA → Done (All tests pass, criteria met) Done → Closed (Released to production)
🔴
To Do
Work not yet started
🟡
In Progress
Currently being worked on
🔵
In Review
Code review or peer review
🟣
In QA
Testing in progress
🟢
Done
All tests passed

Sprint Planning as a QA Engineer

Sprint Planning sets the stage for the entire Sprint. As a QA engineer, your involvement is critical:

Before Sprint Planning

  • Review the Product Backlog items scheduled for the Sprint
  • Ensure acceptance criteria are clear and testable
  • Identify any dependencies that might affect testing
  • Prepare questions for the Product Owner

During Sprint Planning

  • Ask clarifying questions about requirements
  • Provide testing estimates for each story
  • Discuss testability of features
  • Identify integration points that need testing
  • Communicate any risks or concerns early

Story Points vs Testing Effort

Remember: Development story points and QA effort are different. A 3-point story might take 1 point of QA effort, while a 5-point story might require 3 points of testing. Factor this into your estimates.

Daily Scrum for QA

The Daily Scrum is a 15-minute event for the Developers. As a QA engineer, you should:

📊
Report Testing Status
What testing did you complete yesterday?
🎯
Plan Today
What testing will you do today?
🚧
Identify Blockers
Any impediments affecting testing?

Sprint Review

The Sprint Review demonstrates what was accomplished. QA engineers play a key role:

  • Present test results and metrics
  • Demonstrate feature functionality
  • Highlight any known issues or workarounds
  • Discuss testing challenges encountered
  • Provide feedback on product quality

Sprint Retrospective

The retrospective is for the Scrum Team to inspect itself. QA engineers should contribute:

# Example Retrospective Topics for QA What went well? - Test automation coverage increased by 20% - Early collaboration with developers reduced bug count What didn't go well? - Insufficient time for regression testing - Environment issues delayed testing by 2 days Action Items for Next Sprint: - Set up dedicated QA environment by Day 1 - Add 5 more regression test cases to automation suite - Schedule test planning session at sprint start

QA Metrics in Scrum

Tracking metrics helps teams understand their testing effectiveness:

Metric Description Target
Test Coverage Percentage of requirements tested > 90%
Defect Density Bugs per 1000 lines of code < 1.0
Test Execution Rate Tests run per day Trending up
Pass Rate Percentage of tests passing > 95%
Mean Time to Detect Average time to find a bug Decreasing

Agile Testing Quadrants

Lisa Crispin's Agile Testing Quadrants help QA engineers plan comprehensive test strategies:

Q1
Unit Tests
Automated tests that verify individual components. Written by developers.
Q2
Functional Tests
Business-facing tests that verify functionality. GUI, API, integration tests.
Q3
Exploratory Tests
Manual testing to discover edge cases and unusual behaviors.
Q4
Performance Tests
Non-functional testing - load, stress, security, usability.

Key Points

Summary

  • Agile emphasizes iterative development, continuous feedback, and adaptability
  • Scrum provides a framework with defined roles, events, and artifacts
  • QA engineers are integral team members, not a separate phase
  • Definition of Done ensures quality standards are met
  • User stories with clear acceptance criteria enable effective testing
  • Jira workflows help track testing progress through the Sprint
  • Metrics provide insight into testing effectiveness and product quality
