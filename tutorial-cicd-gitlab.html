What is GitLab CI/CD?

GitLab CI/CD is a powerful built-in continuous integration, delivery, and deployment platform that's integrated directly into GitLab. Unlike other CI/CD tools that require separate installation, GitLab CI/CD comes bundled with GitLab, making it seamless to set up and use. It allows you to build, test, and deploy your applications with minimal configuration.

Why GitLab CI for QA Engineers?

  • All-in-One Platform: GitLab combines SCM, CI/CD, and DevOps in one application
  • Auto DevOps: Automatically detect, build, test, and deploy applications
  • Docker Integration: Native Docker support for containerized pipelines
  • GitLab Runners: Scalable, distributed build infrastructure
  • Free Tier: 2,000 CI/CD minutes/month for free accounts

Core Concepts

Let's explore the fundamental building blocks of GitLab CI/CD:

📄
.gitlab-ci.yml
The configuration file that defines your CI/CD pipeline. Placed in the repository root.
🏃
Runners
Agents that execute CI/CD jobs. Can be shared or specific to projects.
🔲
Pipelines
The top-level component containing stages and jobs that run in sequence or parallel.
📦
Jobs
Individual units of work defined in .gitlab-ci.yml. Each job runs in a runner.

Your First GitLab CI Pipeline

Let's create a basic pipeline to understand the structure:

# .gitlab-ci.yml - Your first pipeline # This file defines your CI/CD pipeline # Define stages - jobs run in these stages in order stages: - build - test - deploy # Variables - reusable values across jobs variables: NODE_VERSION: "18" APP_NAME: "my-app" # Build job build: stage: build image: node:${NODE_VERSION} script: - echo "Building ${APP_NAME}..." - npm install - npm run build artifacts: paths: - build/ expire_in: 1 week # Test job - runs in parallel with unit and integration test:unit: stage: test image: node:${NODE_VERSION} script: - npm install - npm run test:unit coverage: /Coverage: \d+\.\d+%/ # Integration tests test:integration: stage: test image: node:${NODE_VERSION} script: - npm install - npm run test:integration services: - postgres:15 variables: POSTGRES_DB: test_db POSTGRES_USER: test POSTGRES_PASSWORD: test # Deploy job - only runs if test stage succeeds deploy: stage: deploy script: - echo "Deploying ${APP_NAME}..." - ./deploy.sh production environment: name: production url: https://example.com only: - main

Advanced Pipeline Features

Matrix Jobs

Run jobs across multiple versions and configurations:

# Matrix strategy - test across multiple configurations test:matrix: stage: test image: node:${NODE_VERSION} script: - npm install - npm test matrix: include: - NODE_VERSION: "16" OS: "ubuntu" - NODE_VERSION: "18" OS: "ubuntu" - NODE_VERSION: "20" OS: "ubuntu" - NODE_VERSION: "18" OS: "windows"

Dependencies & Artifacts

Share build outputs between jobs:

# Build stage creates artifacts build:app: stage: build script: - npm run build artifacts: paths: - dist/ expire_in: 1 day exclude: - dist/node_modules/ # Test stage downloads build artifacts test:e2e: stage: test script: - npm run test:e2e dependencies: - build:app # Deploy stage also gets artifacts deploy: stage: deploy script: - ./deploy.sh dependencies: - build:app

Docker & Services

Use Docker containers for your jobs:

# Using Docker for database testing test:db: image: node:18 services: - name: postgres:15 alias: postgres - name: redis:7 alias: redis variables: POSTGRES_DB: app_test POSTGRES_USER: app POSTGRES_PASSWORD: secret REDIS_URL: redis://redis:6379 script: - npm install - npm run test:db

Running Automated Tests

GitLab CI integrates seamlessly with various testing frameworks:

JUnit Test Reports

# Publishing JUnit test reports test:unit: stage: test image: node:18 script: - npm install - npm test -- --coverage --json --outputFile=report.json artifacts: reports: junit: report.json paths: - report.json expire_in: 1 week

Playwright/Cypress E2E Tests

# Running Playwright E2E tests test:e2e: image: mcr.microsoft.com/playwright:v1.40 stage: test script: - npm ci - npx playwright install --with-deps - npx playwright test artifacts: when: always paths: - playwright-report/ - test-results/ reports: junit: test-results/junit.xml tags: - docker

API Testing with Newman

# Running Postman API tests test:api: image: node:18 stage: test before_script: - npm install -g newman script: - newman run collection.json --environment environment.json --reporters cli,junit --reporter-junit-export newman.xml artifacts: reports: junit: newman.xml when: always

GitLab Runners

Runners execute your CI/CD jobs. They can be hosted by GitLab or self-managed:

☁️
GitLab.com Runners
Hosted runners provided by GitLab. Quick setup, limited minutes.
🖥️
Self-Hosted Runners
Runners on your own infrastructure. Full control, unlimited minutes.
🐳
Container Runners
Run jobs in Docker containers. Great for isolated testing.
☸️
Kubernetes Runners
Auto-scaling runners in Kubernetes clusters.

Registering a Runner

# Install GitLab Runner curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/ | sudo bash sudo yum install -y gitlab-runner # Register a runner sudo gitlab-runner register --url https://gitlab.com --token REGISTRATION_TOKEN --description "my-runner" --tag-list "docker" --executor docker --docker-image "docker:latest" --docker-privileged --docker-volumes "/cache"

Auto DevOps

GitLab Auto DevOps automatically builds, tests, and deploys your application:

# Enable Auto DevOps # Just set this variable in your project settings CI_COMMIT_BRANCH=main # Or in .gitlab-ci.yml variables: DEVOPS_AUTO_DEVOPS_ENABLED: "true" # Auto DevOps will: # 1. Auto Build - Detect language and build # 2. Auto Test - Run appropriate tests # 3. Auto Code Quality - Analyze code # 4. Auto Security - Scan for vulnerabilities # 5. Auto Deploy - Deploy to Kubernetes

Environment Management

Define environments for different deployment stages:

# Deploy to staging deploy:staging: stage: deploy script: - ./deploy.sh staging environment: name: staging url: https://staging.example.com on_stop: stop:staging only: - develop # Deploy to production deploy:production: stage: deploy script: - ./deploy.sh production environment: name: production url: https://example.com on_stop: stop:production when: manual only: - main # Stop environment stop:staging: stage: deploy script: - ./destroy.sh staging environment: name: staging action: stop when: manual
