Behavior-Driven Development with Gherkin

⏱️ Reading Time: 50 min
📊 Difficulty: Intermediate
🔧 Language: Java/Gherkin
📅 Updated: March 2026

Cucumber is a powerful tool that enables Behavior-Driven Development (BDD), bridging the gap between technical and non-technical stakeholders through a shared understanding of software behavior. This tutorial will guide you through creating executable specifications using Gherkin syntax and implementing them with Cucumber.

What is Behavior-Driven Development?

Behavior-Driven Development is an agile software development methodology that encourages collaboration between developers, QA, and business stakeholders. BDD focuses on defining the behavior of a system from the user's perspective using concrete examples.

🔄

The BDD Cycle

BDD follows a three-step iterative cycle: Discuss → Derive → Develop. Teams discuss scenarios in plain language, derive executable specifications from them, and then develop the code to make those specifications pass.

Gherkin Syntax

Gherkin is the business-readable domain-specific language that Cucumber uses to define test cases. It allows you to write specifications in plain English while remaining executable.

Basic Keywords

Keyword Purpose Description
Feature Top-level Groups related scenarios
Background Setup Steps run before each scenario
Scenario Test case A single executable example
Scenario Outline Template Parameterized scenario template
Given Precondition Initial state setup
When Action User action or event
Then Assertion Expected outcome
And / But Extension Chain multiple steps

Your First Feature File

Let's create a complete feature file for a login functionality to understand Gherkin syntax in action:

Feature: User Login Functionality As a registered user I want to log in to my account So that I can access my personalized dashboard Background: Given the user is on the login page And the login form is displayed Scenario: Successful login with valid credentials When the user enters valid username "testuser" and password "password123" And clicks the login button Then the user should be redirected to the dashboard And a welcome message "Welcome, testuser!" should be displayed Scenario: Failed login with invalid password When the user enters username "testuser" and password "wrongpassword" And clicks the login button Then an error message "Invalid credentials" should be displayed And the user should remain on the login page Scenario: Failed login with empty fields When the user clicks the login button without entering credentials Then error messages should be displayed for required fields And the user should remain on the login page

Implementing Step Definitions

Step definitions connect Gherkin steps to executable code. Each step in a scenario maps to a method in your step definition class.

Step Definition Structure

import io.cucumber.java.en.Given; import io.cucumber.java.en.When; import io.cucumber.java.en.Then; import org.junit.Assert; public class LoginStepDefinitions { private WebDriver driver; private LoginPage loginPage; public LoginStepDefinitions() { this.driver = Hooks.driver; this.loginPage = new LoginPage(driver); } // Given steps - Preconditions @Given("the user is on the login page") public void userIsOnLoginPage() { driver.get("https://example.com/login"); } @Given("the login form is displayed") public void loginFormIsDisplayed() { Assert.assertTrue(loginPage.isFormDisplayed()); } // When steps - Actions @When("the user enters valid username {string} and password {string}") public void userEntersCredentials(String username, String password) { loginPage.enterUsername(username); loginPage.enterPassword(password); } @When("the user enters username {string} and password {string}") public void userEntersInvalidCredentials(String username, String password) { loginPage.enterUsername(username); loginPage.enterPassword(password); } @When("clicks the login button") public void userClicksLoginButton() { loginPage.clickLoginButton(); } @When("the user clicks the login button without entering credentials") public void userClicksLoginWithoutCredentials() { loginPage.clickLoginButton(); } // Then steps - Assertions @Then("the user should be redirected to the dashboard") public void userRedirectedToDashboard() { Assert.assertTrue(driver.getCurrentUrl().contains("dashboard")); } @Then("a welcome message {string} should be displayed") public void welcomeMessageDisplayed(String message) { Assert.assertEquals(message, loginPage.getWelcomeMessage()); } @Then("an error message {string} should be displayed") public void errorMessageDisplayed(String expectedError) { Assert.assertTrue(loginPage.getErrorMessage().contains(expectedError)); } @Then("error messages should be displayed for required fields") public void validationErrorsDisplayed() { Assert.assertTrue(loginPage.hasValidationErrors()); } @Then("the user should remain on the login page") public void userRemainsOnLoginPage() { Assert.assertTrue(driver.getCurrentUrl().contains("login")); } }

Using Parameter Types

Cucumber supports converting captured groups in step definitions to specific types using ParameterTypes.

import io.cucumber.java.ParameterType; import java.time.LocalDate; import java.time.format.DateTimeFormatter; public class CustomParameterTypes { // Custom parameter type for dates @ParameterType("\\d{2}/\\d{2}/\\d{4}") public LocalDate date(final String dateString) { DateTimeFormatter formatter = DateTimeFormatter.ofPattern("dd/MM/yyyy"); return LocalDate.parse(dateString, formatter); } // Custom parameter type for prices @ParameterType("\\$\\d+") public Double price(final String priceString) { return Double.parseDouble(priceString.replace("$", "")); } } // Now use in step definitions: @Given("the subscription starts on {date}") public void subscriptionStartsOn(LocalDate startDate) { subscription.setStartDate(startDate); } @When("the user selects the plan costing {price}") public void userSelectsPlan(Double price) { pricingPage.selectPlanWithPrice(price); }

Data Tables in Gherkin

Data tables allow you to pass tabular data directly to step definitions, making scenarios more readable and enabling efficient data-driven testing.

Basic Data Tables

Scenario: Verify user profile fields Given the user profile page is displayed Then the following user information should be correct: | Field | Value | |-------------|----------------------| | Name | John Doe | | Email | john@example.com | | Phone | +1-555-0123 | | Country | Ghana |
import io.cucumber.datatable.DataTable; import java.util.List; import java.util.Map; // DataTable as List of Maps @Then("the following user information should be correct:") public void verifyUserInformation(DataTable dataTable) { List<Map<String, String>> rows = dataTable.asMaps(); for (Map<String, String> row : rows) { String field = row.get("Field"); String expectedValue = row.get("Value"); switch (field) { case "Name": Assert.assertEquals(expectedValue, profilePage.getName()); break; case "Email": Assert.assertEquals(expectedValue, profilePage.getEmail()); break; case "Phone": Assert.assertEquals(expectedValue, profilePage.getPhone()); break; case "Country": Assert.assertEquals(expectedValue, profilePage.getCountry()); break; } } }

Data Table Transformation

Scenario: Add multiple products to cart Given the following products are available: | Product | Price | Quantity | |------------|--------|----------| | Laptop | 999 | 1 | | Mouse | 29 | 2 | | Keyboard | 79 | 1 | When the user adds all products to cart Then the cart should contain 4 items And the total should be $1136
// Transform DataTable directly to List of objects import io.cucumber.java.DataTableType; public class Product { private String name; private Double price; private Integer quantity; // Getters and setters public String getName() { return name; } public Double getPrice() { return price; } public Integer getQuantity() { return quantity; } // DataTable Type entry point @DataTableType public Product productEntry(Map<String, String> entry) { Product product = new Product(); product.name = entry.get("Product"); product.price = Double.parseDouble(entry.get("Price")); product.quantity = Integer.parseInt(entry.get("Quantity")); return product; } } // Use the transformed list directly @Given("the following products are available:") public void productsAvailable(List<Product> products) { for (Product product : products) { inventoryPage.addProduct(product.getName(), product.getPrice()); } }

Cucumber Hooks

Hooks allow you to execute code before and after scenarios, providing setup and teardown functionality.

Types of Hooks

Hook When it runs Use Case
@Before Before each scenario Setup test data, navigate to pages
@After After each scenario Cleanup, screenshot on failure
@BeforeStep Before each step Logging, test data setup per step
@AfterStep After each step Capture screenshots per step
@BeforeAll Before all scenarios Database connection, load config
@AfterAll After all scenarios Close connections, generate reports
import io.cucumber.java.Before; import io.cucumber.java.After; import io.cucumber.java.AfterStep; import io.cucumber.java.Scenario; import org.openqa.selenium.OutputType; import org.openqa.selenium.TakesScreenshot; public class Hooks { private static WebDriver driver; public Hooks(WebDriver driver) { this.driver = driver; } @Before public void setup(Scenario scenario) { System.out.println("Starting scenario: " + scenario.getName()); } @After public void teardown(Scenario scenario) { if (scenario.isFailed()) { // Capture screenshot on failure byte[] screenshot = ((TakesScreenshot) driver) .getScreenshotAs(OutputType.BYTES); scenario.attach(screenshot, "image/png", "failure-screenshot"); } System.out.println("Finished scenario: " + scenario.getName()); } @AfterStep public void afterStep(Scenario scenario) { // Add screenshot after every step (for debugging) if (scenario.isFailed()) { byte[] screenshot = ((TakesScreenshot) driver) .getScreenshotAs(OutputType.BYTES); scenario.attach(screenshot, "image/png", "Step failure screenshot"); } } @Before("@ui-test") public void setupUITest() { driver.manage().window().maximize(); } @After("@ui-test") public void cleanupUITest() { driver.manage().deleteAllCookies(); } }

Organizing with Tags

Tags allow you to organize scenarios and features, enabling selective execution based on criteria like test type, priority, or feature area.

@smoke-test Feature: User Authentication @critical Scenario: Login with correct credentials Given the user is on the login page When the user enters valid credentials Then the user should be logged in successfully @regression Scenario: Password reset flow Given the user clicks on forgot password When the user enters their email Then a reset link should be sent
// Run specific tags from command line mvn test -Dcucumber.filter.tags="@smoke-test" mvn test -Dcucumber.filter.tags="@critical and @regression" mvn test -Dcucumber.filter.tags="@ui-test and not @skip" // In TestNG runner @CucumberOptions( features = "src/test/resources/features", glue = {"com.qa.stepdefinitions"}, tags = "@smoke-test", plugin = {"pretty", "html:target/cucumber-report.html"} ) public class TestRunner {}

Dependency Injection

Cucumber supports dependency injection to share state between step definition classes and manage object creation.

// Using Picocontainer (default) public class Context { public WebDriver driver; public User currentUser; public String currentUrl; } public class LoginSteps { private final Context context; public LoginSteps(Context context) { this.context = context; } @Given("the user logs in with valid credentials") public void userLogsIn() { context.driver.get("https://example.com/login"); context.driver.findElement(By.name("username")).sendKeys("test"); context.currentUser = new User("test"); } } public class DashboardSteps { private final Context context; public DashboardSteps(Context context) { this.context = context; } @Then("the welcome message should show the username") public void verifyWelcomeMessage() { String message = context.driver.findElement(By.className("welcome")).getText(); Assert.assertTrue(message.contains(context.currentUser.getUsername())); } }

Best Practices

Key Takeaways

  • Write Imperative, Not Declarative - Start with detailed steps, then refactor to reusable ones
  • One Assertion Per Scenario - Each scenario should verify one behavior
  • Use Descriptive Names - Scenario names should describe what's being tested
  • Keep Steps Atomic - Each step should be independent and reusable
  • Use Background Wisely - Only for truly common preconditions
  • Tag Strategically - Organize by feature, priority, and test type
  • Business Language - Write Gherkin that business stakeholders can understand
  • Hooks for Infrastructure - Keep hooks minimal; use step definitions for test logic
