Locust Basics

Master Scalable Load Testing with Python

⏱️ Reading Time: 45 min
📊 Difficulty: Beginner
🔧 Language: Python
📅 Updated: March 2026

Welcome to the Locust Basics tutorial! In this comprehensive guide, you'll learn how to use Locust, a powerful open-source load testing tool written in Python. Locust allows you to define user behavior with Python code, making it incredibly flexible and extensible for testing modern web applications.

What is Locust?

Locust is an open-source load testing tool that helps you identify performance bottlenecks in your applications before they impact real users. Unlike traditional point-and-click load testing tools, Locust uses a code-first approach that gives you complete control over your test scenarios.

🐍
Python-Based
Write tests in Python, one of the most popular programming languages
🌐
Distributed Testing
Scale to millions of concurrent users across multiple machines
📊
Real-Time Stats
Monitor performance metrics live during test execution
🔌
Extensible
Extend with custom clients, protocols, and reporting

Installation

Installing Locust is straightforward using pip. Make sure you have Python 3.7 or higher installed before proceeding.

Prerequisites

  • Python 3.7 or higher
  • pip package manager
  • Basic understanding of HTTP protocols
  • A web application to test (we'll use a demo API)

Install Locust

Use pip to install Locust:

pip install locust

Verify the installation:

locust --version

Pro Tip: It's recommended to use a virtual environment to avoid dependency conflicts. Create one with python -m venv venv and activate it with source venv/bin/activate (Linux/Mac) or venv\Scripts\activate (Windows).

Your First Locust Test

Let's create a simple load test to understand the basic Locust concepts. We'll test a sample API endpoint.

Create the Test File

Create a new file named basic_test.py:

from locust import HttpUser, task, between class QuickstartUser(HttpUser): wait_time = between(1, 3) @task def hello_world(self): self.client.get("/hello") @task def view_items(self): for item_id in range(1, 6): self.client.get(f"/item/{item_id}", name="/item/[id]")

Understanding the Code

Let's break down each component:

  • HttpUser - Base class that provides an HTTP client for making requests
  • wait_time - Time in seconds between consecutive task executions (random between 1-3 seconds)
  • @task decorator - Marks a method as a load testing task
  • self.client - The HTTP client used to make requests
  • name parameter - Groups similar requests together in statistics

Run the Test

Start Locust with your test file:

locust -f basic_test.py

Note: By default, Locost starts a web interface on http://localhost:8089. You can also run in headless mode with --headless flag.

HTTP User Tasks

Tasks are the core of your load test. They define what actions your virtual users will perform. Let's explore more advanced task patterns.

Multiple Tasks with Weights

You can control how often each task runs using weights:

from locust import HttpUser, task, between, events class WeightedUser(HttpUser): wait_time = between(1, 3) # This task runs 3 times more often than view_homepage @task(3) def search_products(self): query = ["laptop", "phone", "tablet"] for q in query: self.client.get(f"/search?q={q}") @task(1) def view_homepage(self): self.client.get("/") @task(2) def view_product(self): self.client.get("/products/123")

Sequential Tasks

For scenarios requiring a specific order of operations:

from locust import HttpUser, task, between, SequentialTaskSet class UserBehavior(SequentialTaskSet): # Tasks run in order - login first, then browse, then logout def on_start(self): # Called when user starts - like setup response = self.client.post("/login", json={ "username": "testuser", "password": "testpass" }) if response.status_code == 200: self.token = response.json()["token"] @task def browse_catalog(self): self.client.get("/catalog") @task def add_to_cart(self): self.client.post("/cart", json={ "product_id": "123", "quantity": 1 }, headers={"Authorization": f"Bearer {self.token}"}) @task def on_stop(self): # Called when user stops - like teardown self.client.post("/logout") class AuthenticatedUser(HttpUser): tasks = [UserBehavior] wait_time = between(2, 5)

Advanced Features

Custom Headers and Authentication

Many APIs require authentication tokens or custom headers:

from locust import HttpUser, task, between class AuthenticatedAPIUser(HttpUser): wait_time = between(1, 2) def on_start(self): # Perform login and store token response = self.client.post("/api/v1/auth/login", json={ "email": "user@example.com", "password": "password123" }) self.token = response.json()["access_token"] @property def headers(self): return { "Authorization": f"Bearer {self.token}", "Content-Type": "application/json" } @task def get_user_profile(self): self.client.get("/api/v1/user/profile", headers=self.headers) @task def create_order(self): self.client.post("/api/v1/orders", json={ "items": [{"product_id": "1", "qty": 2}] }, headers=self.headers)

Request Chaining and Data Sharing

Extract data from one request and use it in subsequent requests:

from locust import HttpUser, task, between class EcommerceUser(HttpUser): wait_time = between(2, 4) @task def browse_and_purchase(self): # Step 1: Get product list response = self.client.get("/api/products") products = response.json() # Step 2: Select a random product import random product = random.choice(products) # Step 3: Get product details detail_response = self.client.get( f"/api/products/{product['id']}" ) product_detail = detail_response.json() # Step 4: Add to cart self.client.post("/api/cart", json={ "product_id": product['id'], "price": product_detail['price'], "quantity": 1 }) # Step 5: Checkout self.client.post("/api/checkout", json={ "payment_method": "card" })

Running Load Tests

Command Line Options

Locust provides numerous command-line options for different scenarios:

# Basic usage with headless mode locust -f my_test.py --headless -u 100 -r 10 -t 60s # Options explained: # -u, --users: Number of concurrent users # -r, --spawn-rate: Users to spawn per second # -t, --run-time: How long to run (e.g., 60s, 5m, 1h) # --host: Target host URL # --csv: Save results to CSV files

Common Test Scenarios

# Ramp up test - gradually increase users locust -f test.py --headless -u 500 -r 5 -t 5m --host https://api.example.com # Spike test - sudden increase in users locust -f test.py --headless -u 1000 -r 100 -t 30s --host https://api.example.com # Stress test - find breaking point locust -f test.py --headless -u 5000 -r 50 -t 10m --host https://api.example.com # Endurance test - sustained load locust -f test.py --headless -u 200 -r 2 -t 1h --host https://api.example.com

Best Practices

1. Test Data Management

Use unique test data for each virtual user to avoid false positives from caching:

import uuid @task def create_user(self): unique_email = f"user-{uuid.uuid4()}@test.com" self.client.post("/users", json={ "email": unique_email, "name": "Test User" })

2. Proper Wait Times

Simulate realistic user behavior with appropriate wait times:

# Good: Random wait between realistic user actions wait_time = between(2, 5) # 2-5 seconds between actions # Bad: No wait or too short wait_time = between(0, 0) # Will overload the server

3. Group Similar Requests

Use the name parameter to group dynamic URLs:

# Without grouping - creates separate entries for each item_id self.client.get(f"/items/{item_id}") # With grouping - aggregates under /items/[id] self.client.get( f"/items/{item_id}", name="/items/[id]" )

4. Error Handling

Handle failures gracefully and log important information:

from locust import events @events.request.add_listener def on_request(request_type, name, response_time, response_length, exception, **kwargs): if exception: print(f"Request failed: {name} - {exception}")

Key Points Summary

  • Locust is a Python-based load testing tool with distributed testing capabilities
  • Define user behavior with @task decorated methods in Python
  • Use HttpUser class for HTTP-based testing
  • Control task frequency with weights on @task decorators
  • Use SequentialTaskSet for ordered task execution
  • Implement on_start() for setup and on_stop() for teardown
  • Use the name parameter to group dynamic URLs in statistics
  • Always use unique test data to avoid false positives
  • Ramp up users gradually to observe system behavior under load
